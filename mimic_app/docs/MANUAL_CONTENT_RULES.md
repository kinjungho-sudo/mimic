# 매뉴얼 제목/본문 생성 규칙

이 문서는 Parro가 녹화 데이터를 매뉴얼로 바꿀 때 `tutorial_title`, `user_title`, `user_script`, annotation label을 어떻게 작성해야 하는지 정의한다.

기존 문서나 프롬프트에 `~를 클릭합니다` 중심 규칙이 남아 있더라도, 매뉴얼 콘텐츠 품질 판단은 이 문서를 우선한다.

## 기본 원칙

매뉴얼은 사용자가 클릭한 좌표를 설명하는 문서가 아니라, 사용자가 따라 해서 업무를 완료하도록 돕는 문서다.

- 제목은 **목적**을 말한다.
- 본문은 **왜/무엇을/어떤 상태로** 안내한다.
- annotation label은 **위치나 대상**을 짧게 표시한다.

예를 들어 `gemini.google.com 주요 영역을 클릭합니다.`는 실패한 문장이다. 사용자가 실제로 하려는 일이 프롬프트 작성이라면 다음처럼 작성한다.

- 전체 제목: `Gemini로 답변 생성하기`
- 스텝 제목: `프롬프트 입력 준비`
- 본문: `Gemini에게 요청할 내용을 작성할 수 있도록 프롬프트 입력 영역을 선택합니다.`
- annotation label: `입력창 선택`

## 전체 제목 규칙

`tutorial_title`은 첫 클릭이나 첫 화면 이름이 아니라, 전체 절차가 끝났을 때 사용자가 달성하는 최종 목적을 쓴다.

제목을 만들기 전에 전체 단계를 다음 순서로 해석한다.

1. 연속된 클릭·입력을 `시작`, `설정`, `완료/검증` 같은 업무 구간으로 묶는다.
2. 마지막 단계에서 사용자가 얻거나 확인하는 상태를 찾는다.
3. 첫 단계가 아니라 마지막 결과를 제목의 핵심 동사로 삼는다.
4. 생성→설치→테스트처럼 목적이 여러 구간에 걸치면 마지막 구간까지 제목에 포함한다.
5. DOM 라벨, ID, 버튼명은 목적을 추론하는 증거로만 사용하고 제목에 그대로 복사하지 않는다.

제목은 필요하면 두 개의 목적 동사를 연결할 수 있다.

- `만들고 테스트하기`
- `작성 후 보내기`
- `설정하고 게시하기`
- `연결하고 동작 확인하기`

좋은 제목:

- `Gemini로 답변 생성하기`
- `메일 작성 후 보내기`
- `Slack 채널 만들기`
- `상품 주문 완료하기`

나쁜 제목:

- `메일 클릭하기`
- `gemini.google.com 주요 영역 클릭`
- `버튼 클릭하기`
- `받은편지함(3388) 클릭`
- `Slack에 앱 추가하기` — 앱 생성·권한 설정·설치·테스트 중 어디까지 하는지 알 수 없음

### 여러 구간이 있는 매뉴얼 예시

다음 흐름을 하나의 매뉴얼로 녹화했다고 가정한다.

`새 Slack 앱 생성 → 매니페스트 입력 → 권한·토큰 설정 → 워크스페이스 설치 → 에이전트에게 질문해 응답 확인`

- 좋은 전체 제목: `Slack AI 에이전트 앱 만들고 테스트하기`
- 허용 가능한 축약: `Slack 앱을 만들고 워크스페이스에 설치하기` — 테스트 단계가 매뉴얼 범위에서 제외된 경우
- 나쁜 전체 제목: `Slack에 앱 추가하기`, `Create New App 클릭하기`, `OAuth 설정하기`

전체 흐름에 테스트·전송·게시·구매·설치 같은 완료 단계가 있으면 제목도 해당 최종 상태를 표현해야 한다. 중간 단계만 설명하는 제목은 다시 생성한다.

Notion에서 `페이지 선택 → 스케줄 선택 → 일정 내용 입력` 흐름이 기록되었다면 제목은 `app.notion.com 주요 영역 선택하기`가 아니라 `Notion에 일정 기록하기`로 작성한다. Claude 호출이 실패한 fallback에서도 도메인보다 스케줄 작성 목적을 우선한다.

## 스텝 제목 규칙

`user_title`은 한 단계의 업무 목적을 20자 안팎으로 요약한다. 화면에 보인 텍스트를 그대로 베끼지 말고, 사용자가 이해할 말로 바꾼다.

좋은 제목:

- `프롬프트 입력 준비`
- `메일함 확인`
- `메일 본문 작성`
- `입력 내용 적용`
- `보내기 실행`

나쁜 제목:

- `gemini.google.com 주요 영역 클릭`
- `메일, 읽지 않은 메일 2458개 클릭`
- `프롬프트 입력 입력`
- `5 38fcaf5650aa8020868f... 클릭`
- 긴 이메일 본문이나 프롬프트 원문 전체

## 본문 규칙

`user_script`는 사용자가 그대로 따라 할 수 있는 한 문장 중심의 안내문으로 쓴다.

기본 구조:

`왜 하는지` + `무엇을 하면 되는지` + `그 다음 어떤 상태가 되는지`

좋은 본문:

- `Gemini에게 요청할 내용을 작성할 수 있도록 프롬프트 입력 영역을 선택합니다.`
- `받는 사람에게 보낼 내용을 작성하기 위해 메일 본문 입력란에 내용을 입력합니다.`
- `작성한 내용이 적용되도록 입력을 완료합니다.`

나쁜 본문:

- `gemini.google.com 주요 영역을 클릭합니다.`
- `감사하다는 내용과 함께...`처럼 사용자가 입력한 긴 원문을 그대로 복사한 문장
- `프롬프트 입력 입력합니다.`처럼 중복된 동작어
- UUID, hash, DOM id, aria-label, 읽지 않은 개수 같은 기계적 정보가 섞인 문장

## Annotation Label 규칙

annotation label은 매뉴얼 문장이 아니라 화면에서 어디를 봐야 하는지 알려주는 짧은 표식이다.

좋은 label:

- `입력창 선택`
- `본문 입력`
- `보내기`
- `저장`
- `메뉴 열기`

나쁜 label:

- `gemini.google.com 주요 영역 클릭`
- 긴 프롬프트 원문
- 긴 이메일 본문
- `입력 입력`

## 불확실할 때의 fallback 규칙

AI가 화면 맥락을 충분히 이해하지 못하면 억지로 클릭 대상을 만들지 않는다.

- 클릭 대상이 불명확하면 `주요 영역 클릭` 대신 `화면 확인`, `입력 영역 선택`, `다음 단계 준비`처럼 중립적으로 쓴다.
- URL이나 도메인은 맥락 힌트로만 사용하고 제목/본문의 주어로 직접 쓰지 않는다.
- 원문 텍스트가 길거나 민감할 수 있으면 목적 중심으로 요약한다.
- 사용자의 입력 내용은 원문 복사보다 `요청 내용 입력`, `메일 본문 입력`, `입력 내용 적용`처럼 역할로 표현한다.

## 구현 계약

다음 코드를 수정할 때는 이 문서의 기준을 함께 반영해야 한다.

- `lib/ai/claude.ts`: GPT-5.6 Luna 프롬프트가 캡처 화면과 전체 작업 흐름을 근거로 목적 중심 제목/본문을 생성해야 한다.
- `lib/ai/capture-fallback.ts`: fallback 문장이 `대상 + 클릭합니다`로 되돌아가지 않아야 한다.
- `scripts/verify-capture-fallback.js`: `gemini.google.com 주요 영역 클릭`, `프롬프트 입력 입력`, 긴 입력 원문 복사 같은 회귀 케이스를 검증해야 한다.

GPT-5.6 Luna가 만든 전체 제목과 모든 단계의 제목/본문은 저장 전에 한꺼번에 품질을 검사한다. 내부 도메인, 범용 문구, 과도한 중복, 누락 단계가 하나라도 있으면 어떤 단계도 저장하지 않고 기존 내용을 유지한다.

검증은 build 성공만으로 끝내지 않는다. 새로 생성한 매뉴얼을 사람이 읽었을 때 실제 업무 매뉴얼처럼 보이는지 확인한다.
